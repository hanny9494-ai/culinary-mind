PR #22 修复方向正确——四项 F1-F4 完整覆盖了 P0 root cause（缺 `task_id` 直接 unlinkSync 删除证据）。但当前实现遗留两个 P1 问题，与"证据不丢失"和"避免重复处理"直接相关，建议合入前修：

## D. 必修清单（合入前必须完成）

### P1（blocker）

1. **唯一归档路径**：`_orphan_no_task_id/` 与 `_processed/` 路径都不能仅用 `basename(filePath)`。  
   建议实现：
   ```ts
   function uniqueArchivePath(dir: string, name: string): string {
     return join(dir, `${Date.now()}_${process.pid}_${Math.random().toString(36).slice(2,8)}_${name}`);
   }
   ```
   两处使用：line 207（orphan）和 line 269（processed）。

2. **renameSync 失败 → 不要重复处理**：至少满足以下一点：
   - 短重试（EBUSY/EAGAIN，2-3 次，间隔 100-500ms）
   - 失败时写 `_archive_failed/<source>.failed.json` marker，next poll 看到 marker 跳过 `appendRaw` + 业务流程
   - 在 `processingResults` Set 中保留失败文件的标记（按时间过期）
   - 业务幂等：`appendRaw` 可加 `source_inode + mtime` 字段做去重，下游 wiki compiler 处理时按此字段 dedup

3. **测试补充**至少这 3 条以锁定 P1 修复行为：
   - `archive: duplicate basename does not overwrite existing orphan`
   - `archive failure: does not append raw repeatedly`
   - `archive failure: file is quarantined or marker is written`

### P2（建议本 PR 顺手做）

4. `firstNonEmptyString()` helper 替代 `||` 链（防 `0` / `{}` / `""` 误用）。
5. `data.to` 缺失且 task 不在 DB 时：`console.warn` + `createEvent({type: 'result_undeliverable', ...})`。
6. 测试补：`ref_task` / `dispatch_id` fallback、优先级（`task_id` 与 `ref_dispatch` 同时存在时使用 `task_id`）、空字符串、非字符串字段。

### Info / Follow-up（不阻塞合入）

7. `_processed/` 与 `_orphan_no_task_id/` 的 GC / 滚动策略——单独 issue，可观察 1 周后定量评估。
8. handleDispatch 也加 `_processed_dispatches/` 归档（对称性）。
