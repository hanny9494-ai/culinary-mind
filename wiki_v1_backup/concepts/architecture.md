# concepts — architecture


## Updates (2026-04-04)
- L0 layer (科学原理图谱): Scientific principles graph, causal chains + parameter boundaries + 17 domains. Status: ✅ 50,000+ entries (wrapping up). L0 is the 'referee' — all recipes, external data, and substitution suggestions must pass L0 principle validation.
- L1 layer (设备实践参数层): Equipment practice parameters — how to adjust the same principle for different equipment. Status: ⏳ Not yet built.
- L2a layer (天然食材参数库): Natural ingredient parameter library — variety/cut/season/origin/price. Status: ⏳ Not yet built (Pilot with 75 varieties completed).
- L2b layer (食谱校准库): Recipe calibration library — validated parameter combinations + credibility scores. Status: ✅ 29,085 recipes from 63 books.
- L2c layer (商业食材数据库): Commercial ingredient database — brand/model → component breakdown. Status: ⏳ Not yet built.
- FT layer (风味目标库): Flavor target library — aesthetic terms → quantifiable sensory parameters. Status: ⏳ Not yet built.
- L3 layer (推理引擎): Reasoning engine — pre-computation + real-time inference. Status: ⏳ Not yet built.
- L6 layer (翻译层): Translation layer — Cantonese cuisine language ↔ system language. Status: ⏳ Not yet built. Principle: L6 only translates, does not judge.
- 17 scientific domains in L0: protein_science, carbohydrate, lipid_science, fermentation, food_safety, water_activity, enzyme, color_pigment, equipment_physics, maillard_caramelization, oxidation_reduction, salt_acid_chemistry, taste_perception, aroma_volatiles, thermal_dynamics, mass_transfer, texture_rheology. Domain count is fixed — out-of-domain principles temporarily marked 'unclassified', 17 domains do not expand.
- Agent directory location: .claude/agents/ — CC Lead scans this directory on startup to discover available agents. New agents created by adding .md files with frontmatter (name/description/tools/model) and system prompt. No changes to CLAUDE.md required.
- Project root directory: ~/culinary-engine. All code, config, docs, agents in this repo. output/ is a symlink → ~/l0-knowledge-engine/output (data not in git). STATUS.md at ~/culinary-engine/STATUS.md (single source of truth).
- Architectural principle: L0 is the referee — recipes and external data must pass L0 validation. L6 only translates, does not judge. Out-of-domain principles temporarily marked 'unclassified' — 17 domains do not expand.
- Architecture document location: docs/culinary_engine_architecture_v5.docx (七层知识架构 v5)
- OCR tool recommendation: PaddleOCR/MinerU for text extraction + Flash VLM for image recognition
- CC Lead infrastructure migrated to new repo: CLAUDE.md + .claude/agents/ + scripts/dify/
- L0 pipeline Stage progression: Stage1 (extraction) → Stage2 (matching, 306 questions) → Stage3 (distillation/atomic propositions) → Stage3B (splitting) → Stage4 (net increase)
- Architecture principle: L0 acts as the judge/referee (裁判) for L2b recipe calibration library
- Architecture principle: 7-layer knowledge architecture (L0, L1, L2a, L2b, L2c, FT, L3, L6) — note L3 and L6 are named layers, FT is a separate named layer
- Embedding model for Stage2 matching: Ollama qwen3-embedding:8b
- API routing rules: qwen* → DashScope, claude* → 灵雅 proxy, local models → Ollama localhost:11434, gemini* → Google API or 灵雅 proxy
- L0-L2 linking uses Three-Hop architecture: (Ingredient)-[:CONTAINS {concentration}]->(Compound)-[:GOVERNED_BY]->(ScientificPrinciple) and (ProcessStep)-[:TRIGGERS {confidence}]->(ScientificPrinciple) and (ProcessStep)-[:TRANSFORMS {from_state, to_state}]->(Compound)
- Two parallel L0-L2 linking mechanisms: A) Pre-computed static edges from FoodAtlas+USDA import (reliable, deterministic, for known ingredient science); B) Vector similarity via Neo4j vector index on L0 embeddings (broad coverage, supports novel combos, runtime discovery)
- Neo4j vector index for L0 embeddings: CREATE VECTOR INDEX l0_embedding FOR (p:ScientificPrinciple) ON (p.embedding) with vector.dimensions=1024 and vector.similarity_function='cosine'. Query threshold: score > 0.75, filter by node.domain IN $relevant_domains
- USDA fdcId is precise to body-part + processing-state level (e.g., raw chicken breast vs cooked chicken leg are different fdcIds). One L2a atom (e.g., 'chicken') maps to MULTIPLE fdcIds, requiring a mapping table not 1:1 binding.
- USDA FDC Neo4j pattern: (:Ingredient {atom_id})-[:HAS_USDA_ENTRY {part, state}]->(:UsdaFood {fdcId, description}). USDA data is NOT embedded in atom node but linked via relationship edges. L2c commercial ingredients use same pattern with USDA Branded.
- FoodAtlas Neo4j linking pattern: (:Ingredient)-[:CONTAINS {confidence, source:'foodatlas'}]->(:Compound)-[:GOVERNED_BY {domain}]->(:ScientificPrinciple)
- Recipe Schema v1 follows ISA-88 three-segment separation: process (action sequence), formula (ingredients + quantities + params), equipment (device list) — each segment changes independently
- Recipe is the assembly layer: references SubRecipes via components[], main_ingredients[], garnish[], refs[] — four distinct ingredient source types
- Neo4j schema design: Principle node {id, domain, proposition, confidence: high/medium/inferred, source_book, causal_chain: Boolean}; Recipe node {id, name, name_zh, book, cuisine}; Ingredient node {name, name_zh, foodon_id}; Domain node (17 pre-built); relationships: Recipe-[:GOVERNED_BY {confidence, decision_point}]->Principle, Principle-[:BELONGS_TO]->Domain
- L0 has 17 domains — Domain nodes are pre-built in Neo4j
- Inference engine (L3) Plan C: neo4j-graphrag-python as retrieval layer + LangGraph StateGraph as reasoning agent graph
- Graphiti integration: native Neo4j support confirmed, mature tutorials for LangGraph + Graphiti long-term memory. Used for L3-personal: recording user taste preferences, query history, ingredient blacklists
- SubRecipe Registry JSON structure: {SR-001: {name, name_zh, source_book, source_chunk_id, aliases: [], referenced_by: [RCP-xxx, ...]}}. Aliases support multiple name variants for matching
- Process field future upgrade path: currently linear step list with parallel_group field reserved. P4 upgrade: process → DAG supporting parallel/merge/time optimization/equipment conflict detection
- Three-pass scanning strategy for cross-recipe reference graph: Pass 1 = identify all 'output nodes' (SubRecipe definition locations); Pass 2 = identify all 'reference edges' (where recipes mention SubRecipes); Pass 3 = connect output nodes + reference edges = cross-recipe reference graph
- L6 is a translation layer for Cantonese language conversion, positioned after Claude synthesis in the L3 response pipeline
- Ingredient node in Neo4j includes foodon_id field for FoodOn ontology alignment
- Recipe-[:GOVERNED_BY]->Principle relationship has properties: confidence (string) and decision_point (string)
- source_chunk_id format in SubRecipe Registry: {book_name}:chunk_{number}, e.g., professional_baking:chunk_2801
- Recipe schema source field: {book: string, page: null} at Recipe level; {book: string} at SubRecipe level
- sub_components field in SubRecipe schema: [{ref: SR-xxx, role: string}] — allows SubRecipe to reference other SubRecipes (nested sub-recipes)
