## 数据存储性能架构

### 分层存储策略

不把所有数据塞进一个库。按查询频率分三层：

**热层（Neo4j 主库，毫秒级响应）：**
- L0 原子命题（5,000-8,000 节点）
- 配方 + SubRecipe（500-2,000 节点）
- 配方×L0 映射边（~10,000-30,000 边）
- 食材标准名映射（FoodOn 核心子集 ~10,000 条）
- 风味化合物关系（FlavorGraph 核心子集）
- 预估：~50,000 节点 + ~100,000 边，单次查询 <10ms

**温层（Weaviate，语义检索，~50ms）：**
- 所有 embedding 向量（chunks + 原理 + 配方）
- 用于自然语言查询和相似度搜索

**冷层（按需查询，可以略慢）：**
- FoodKG 完整数据（6700万三元组，USDA 营养详情）
- FoodAtlas 23万化学关系
- Recipe1M 百万配方参考
- 可用独立 PostgreSQL / Neo4j 冷实例
- 单次查询 100-500ms，只在需要营养/化学详情时调用

**查询策略：**
- 日常推理（"这道菜怎么调"）→ 只查热层 + 温层
- 需要营养详情（"这道菜的蛋白质含量"）→ 查冷层 USDA
- 需要化学成分（"这个香料的萜烯类型"）→ 查冷层 FoodAtlas
- LLM 推理时间 ~2-5秒，数据库不是瓶颈
